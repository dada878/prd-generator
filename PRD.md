# 🧠 PRD：需求確認生成器（Requirement Clarity Assistant）

## **C — Context（產品情境）**

* **背景**：在使用 AI（如 ChatGPT、Claude）進行「Vibe Coding」或產品構想時，使用者常以一句模糊描述開始，導致 AI 猜錯方向、開發結果偏離預期。
* **問題**：

  * 使用者輸入的需求過於模糊，AI 難以理解實際意圖。
  * 傳統對話式 AI 缺乏系統性「反問」機制。
  * 需求澄清過程耗時，且結果難以轉換成正式 PRD。
* **解法**：建立一個能根據模糊需求主動「提問 → 選項澄清 → 回答收集 → 生成 PRD」的互動系統。
* **主要使用者**：

  * 產品經理、設計師、創業者、開發者。

---

## **A — Appearance（設計風格）**

* **風格**：對話式、乾淨、專注，避免資訊干擾。
* **配色**：主色藍、輔色灰白，與 Chat UI 一致。
* **互動方式**：

  * Chat 型介面。
  * 問答框支援：

    * 文字回答。
    * 預設選項（Button chips）。
  * 顯示「需求生成進度條」或「澄清階段」。

---

## **S — Structure（流程結構）**

### 1️⃣ 初始輸入階段

* 使用者輸入一句模糊需求，例如：

  > 「我想做一個幫人管理任務的 app」

* 系統偵測模糊程度（可用 LLM prompt 分析關鍵元素缺失）。

* 進入「問題生成階段」。

---

### 2️⃣ 問題生成階段

* 根據需求類型自動產生一組澄清問題，分層設計：

  1. **背景層**：你要解決什麼問題？誰會用？
  2. **功能層**：你希望用戶可以做什麼？
  3. **互動層**：這應該是聊天、表單還是看板？
  4. **輸出層**：最終你希望 AI 幫你產出什麼？（PRD、設計稿、User Flow…）

* 每個問題都：

  * 提供 2–4 個快速選項（button）。
  * 可改用文字輸入。

---

### 3️⃣ 回答收集階段

* 使用者逐步回答問題。
* 系統自動彙整為「明確需求結構」。
* 同步顯示一份「即時更新」的 PRD 草稿（例如在右側）。

---

### 4️⃣ 生成與審閱階段

* 當使用者完成主要問題後：

  * 系統生成完整 PRD（可用 CAST 或標準 PRD 格式）。
  * 顯示一個「二次審閱問題列表」，例如：

    * 是否有設定優先順序？
    * 是否有提到技術邊界？
    * 是否需要定義非功能需求？

* 使用者可回覆或略過，最後生成最終版 PRD。

---

### 5️⃣ 輸出階段

* 可選擇：

  * 下載為 Markdown / PDF。
  * 匯出至 Notion / Linear / Jira。
  * 儲存至專案資料夾。

---

## **T — Tech Stack（技術架構）**

* **前端**：Next.js + Shadcn UI
* **AI 核心**：GPT-5（語言模型）

  * 任務模組：

    * 模糊需求解析器（Intent Detector）
    * 問題生成器（Question Generator）
    * 需求整合器（PRD Synthesizer）
* **後端**：Firebase Admin SDK + Firestore

  * 儲存使用者 Session、問題回答紀錄、PRD 草稿
* **登入／權限**：Auth.js + Google OAuth
* **部署**：Zeabur
